## 🔍 코드 리뷰

### 잘 된 점

- `useEditingStore`에서 탭·필터·검색어를 전역 관리 — 라우팅 이동 후 복귀해도 상태 유지 (`EditingListPage.tsx:67-73`)
- `useMemo` 로 탭필터 → 소싱처추출 → 최종필터 → 정렬을 단계적으로 분리해 불필요한 재계산 최소화 (`EditingListPage.tsx:155-187`)
- `CollectionProgressBanner`가 `useSourcingStore` 직접 구독 — 부모 컴포넌트 의존 없이 독립적으로 동작
- 번역 완료 후 상품명 전환 우선순위 (`PENDING_JA_TITLES → p.titleJa → toJaTitle()`) — 새로 수집된 상품도 누락 없이 처리 (`useEditingStore.ts:198`)

## 📝 의사결정 기록

### 재번역 시 탭 이탈 방지
- **결정**: 재번역 시 `translationStatus`를 `'processing'`으로 변경하지 않고 `isReTranslating: true` 플래그 추가
- **고려한 대안**: `translationStatus = 'processing'`으로 변경 → 번역완료 탭에서 항목이 사라짐
- **선택 이유**: 이미 번역된 상품이 탭에서 사라지는 UX가 혼란스러움
- **트레이드오프**: `isProcessing` 계산 로직이 두 가지 조건을 처리해야 함

### MOCK 더미 데이터 제거
- **결정**: `products: []` 초기값, 소싱으로 수집한 상품만 표시
- **고려한 대안**: 48건 MOCK 데이터 유지 (최초 계획)
- **선택 이유**: UT에서 실제 소싱 → 편집 플로우 검증이 목적
- **트레이드오프**: 빈 상태에서 시작하므로 소싱을 먼저 해야만 편집 화면 검증 가능

### 번역 완료 후 일본어 제목 결정 우선순위
- **결정**: `PENDING_JA_TITLES[id] → p.titleJa → toJaTitle(titleKo)` 3단계 폴백
- **고려한 대안**: `toJaTitle()` 단일 처리
- **선택 이유**: 소싱 시점에 `titleJa`를 저장하므로 해당 값 우선 사용. PENDING 테이블은 기존 MOCK 호환용
- **트레이드오프**: 세 가지 소스가 혼재 — 실제 API 연동 시 단순화 필요

### setTimeout × N (setInterval 금지)
- **결정**: 상품 수만큼 `setTimeout` 1회씩, 누적 딜레이 적용
- **고려한 대안**: `setInterval`로 순차 처리 (소싱 파트에서 이중 알림 버그 발생한 방식)
- **선택 이유**: 계획 문서에 명시적으로 `setInterval` 금지 지시
- **트레이드오프**: 딜레이 계산이 번역 시작 시점에 고정됨
