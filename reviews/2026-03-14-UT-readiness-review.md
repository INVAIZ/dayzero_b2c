> **작성일**: 2026-03-14
> **대상**: DayZero B2C 전체 화면 — UT(사용성 테스트) 충족 여부 분석
> **기반 문서**: `docs/02_UX 설계 기준.md`, `docs/03_IA 구조 설계.md`, `docs/04_화면별 기능 명세서.md`, `docs/05_프로젝트 완료 기준.md`

## 📋 한눈에 보기

> 기술 용어 없이 설명합니다.

DayZero의 핵심 UT 목표인 **"초보 셀러가 소싱 → 편집 → Qoo10 등록까지 막힘 없이 완주할 수 있는가"** 를 기준으로 평가했을 때, 이 경로의 주요 화면은 모두 만들어져 있습니다. 특히 등록 이후 가격·재고 변동 모니터링까지 구현된 것은 MVP 범위를 초과하는 수준입니다.

다만 IA에서 설계한 **"내 상품(PM-01~03)" 섹션이 별도로 존재하지 않고**, 현재 "등록" 메뉴가 그 역할까지 겸하고 있어 테스터가 "등록한 상품을 나중에 어디서 관리하나요?"라고 혼란을 겪을 수 있는 지점이 남아있습니다. 설정(ST) 화면도 없으나 UT 범위 밖이므로 문제가 되지 않습니다.

**전체 UT 준비도**: `████████░░` **78%** — 핵심 플로우 완성, 일부 연결고리 보완 필요

## ⚠️ UT 시나리오별 리스크 체크

UT에서 가장 많이 쓸 시나리오 3가지 기준으로 점검했습니다.

### 시나리오 A: 신규 셀러 첫 상품 등록
```
로그인 → 온보딩(S-01~04) → URL 소싱(SC-02) → 편집(ED-01~02) → 등록 → RG-02
```
- ✅ 전체 경로 라우팅 연결됨
- ✅ 각 단계 CTA 버튼 존재
- ⚠️ 온보딩 완료 후 SC-01로 자동 이동하는지 확인 필요

### 시나리오 B: 등록된 상품 모니터링
```
등록 탭 진입 → 판매 중 탭 → 변동 확인 중 탭 → 품절 상품 일시 중지 → 판매 중지 탭 확인
```
- ✅ 3탭 구조 완성
- ✅ 판매 일시 중지 → 판매 중지 탭 이동 동작
- ✅ 판매 재개 + 품절 경고 모달 동작

### 시나리오 C: 등록 실패 상품 재처리
```
RG-02b 실패 탭 → 에러 사유 확인 → "수정 후 재등록" → ED-02 → 재등록
```
- ⚠️ 실패 탭 → ED-02 이동 버튼 동작 직접 확인 필요 (코드 경로 불확실)

## 📝 주요 의사결정 기록 (이번 세션)

### RG-02 = PM-01 통합 결정
- **결정**: "내 상품" 별도 섹션 없이 "등록" 라우트(`/registration`)가 판매 중인 상품 전체 관리를 담당
- **고려한 대안**: PM-01~PM-03 별도 구현
- **선택 이유**: UT 프로토타입 범위에서 화면 수를 최소화, 핵심 플로우 완성 우선
- **트레이드오프**: GNB "등록" 라벨이 등록 전/후 모두를 의미하게 되어 IA와 불일치

### 3탭 구조 (판매 중 / 변동 확인 중 / 판매 중지)
- **결정**: 전체 탭 제거, 판매 중 기본 탭, 이슈 상품은 변동 확인 중 탭 내 상단 정렬
- **고려한 대안**: 4탭(전체·변동확인중·역마진품절·판매중지)
- **선택 이유**: 탭이 많을수록 초보 셀러 인지 부하 증가. 이슈는 뱃지로 수량 표시
- **트레이드오프**: "역마진·품절만 보기" 필터가 없어짐 → 변동 확인 중 탭 내 상단 정렬로 보완

### 판매 중지 상품의 모니터링 유지
- **결정**: 판매 중지 상품도 monitoring.status를 active로 유지 (이슈 카운트에서만 제외)
- **선택 이유**: 품절 상품이 재입고됐을 때 셀러가 알 수 있어야 판매 재개 결정 가능
- **트레이드오프**: "판매 중지 = 관리 안 함"으로 기대할 수 있는 셀러 혼란 가능성
