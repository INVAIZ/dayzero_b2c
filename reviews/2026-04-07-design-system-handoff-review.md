## 코드 리뷰

### 잘 된 점

- **단일 소스 원칙 확립**: `tokens.ts` 하나에서 모든 디자인 값을 관리. 55개 파일이 이 파일을 import.
- **일관된 import 경로**: 모든 55개 파일이 상대경로(`../../design/tokens`)를 사용. `@/` alias 없음.
- **토큰 네이밍 일관성**: 기존 패턴(`primary/primaryDark/primaryLight`)을 따라 orange 계열, scheduled 계열 추가.
- **문서 구조화**: 토큰 → 패턴 → 컴포넌트 API → 핸드오프 순서로 계층적 참조 가능.
- **빌드 무결성**: 260건+ 치환 후 TypeScript 에러 0건.

### 개선이 필요한 점

## 의사결정 기록

### 근사값 정규화 vs 완전 보존
- **결정**: 시각적 차이가 인지 불가한 근사값은 기존 토큰으로 정규화
- **고려한 대안**: A) 모든 고유 hex에 토큰 추가 vs B) 근사값 정규화
- **선택 이유**: A안은 토큰 수 폭증 (10개+)으로 관리 복잡도 증가. 실제 렌더링 차이 없음
- **트레이드오프**: 원본 hex 값과 1~8 RGB 단위 미세 차이 발생. 육안 구분 불가

### 데모 목업 폰트 크기 유지
- **결정**: ExtensionInstallPage의 7/8/9px 폰트를 토큰화하지 않고 유지
- **고려한 대안**: 토큰 추가 vs 하드코딩 유지
- **선택 이유**: 이 크기는 실제 UI가 아닌 쇼핑몰 시뮬레이션 목업용. 디자인 시스템 범위 밖
- **트레이드오프**: 3개 사이즈가 토큰 외 하드코딩으로 남음

### 문서 분리 구조
- **결정**: 단일 문서가 아닌 4개 파일로 분리 (tokens.ts, DESIGN_SYSTEM.md, COMPONENT_API.md, HANDOFF.md)
- **고려한 대안**: 하나의 거대 문서 vs 역할별 분리
- **선택 이유**: 각 문서의 독자와 목적이 다름. 개발자는 필요한 것만 빠르게 참조 가능
- **트레이드오프**: 문서 간 중복 참조 발생 (HANDOFF → DESIGN_SYSTEM → COMPONENT_API)
