# UX 설계 기준

> 이 문서는 DayZero B2C 프로토타입의 UX 설계 원칙을 정의합니다.
> 3회 사용성 테스트(UT)를 거쳐 검증·보완된 기준입니다.
> 모든 화면 구현 시 이 원칙을 준수해야 합니다.

---

## 배경

### 경쟁사 공통 문제
투플렉스, 셀러픽, 윈들리, 퍼센티, 셀잇파파, Q10 Auto 등 6개 경쟁사가 공통으로 겪는 문제:
- **정보 과부하**: 소싱/가공/등록/재고 데이터를 한 화면에 과도하게 노출
- **단계의 복잡성**: 셋팅→소싱→가공→등록→싱크 각 단계에 세부 단계가 포함되어 인지 부하 누적
- 초보자가 "다 좋은데 어디서부터 해야 할지 모르겠다"는 상황에서 이탈

### 설계 목표
1. **초보 셀러도 첫 상품 등록까지 막힘 없이 도달** — 가입 후 첫 등록 완료까지 최소 단계
2. **반복 조작을 줄여 시간 절약** — AI 자동 완성 + 대량 처리로 반복 최소화

## 4대 설계 원칙

### 원칙 1: 지금 해야 할 것만 강조

> 선택지가 많아지면 초보자는 멈춘다.

- 화면당 핵심 액션 3개 이하
- 부수적 설정은 숨기고 기본값으로 자동 처리
- 작업 흐름을 단계별로 명확하게 분리

**프로토타입 적용 사례:**
- 온보딩(S-02~S-05): 한 화면에 한 가지 설정만 (Qoo10 연동 → 기본 설정 → 프로그램 설치)
- 자동 수집(SC-03): 3단계 위저드 (쇼핑몰 → 카테고리 → 시간)
- 편집 상세(ED-02): 기본정보/가격/이미지를 탭으로 분리

### 원칙 2: 시스템 선처리, 셀러 최종 확인

> 셀러가 판단하기 어렵거나 반복적인 작업은 시스템이 먼저 처리한다.

**자동화 3단계:**
| 수준 | 설명 | 예시 |
|------|------|------|
| AI 자동 완성 | 시스템이 알아서 채움, 셀러는 확인만 | 카테고리 매핑, 상품명 번역, 가격 환산 |
| 스마트 기본값 | 시스템이 추천하고 셀러가 조정 | 마진율 30%, 배송비 대행사 기반 자동 |
| 셀러 직접 입력 | 셀러만 알 수 있는 정보 | 커스텀 설명, 특이사항 |

**프로토타입 적용 사례:**
- 배송대행사 선택 시 창고 주소 자동 입력 (S-03)
- AI 번역 원클릭 (ED-02): 상품명, 설명, 옵션 자동 번역
- 가격 자동 계산: 원가→마진→수수료→판매가 자동 산출
- 카테고리 AI 자동 매핑 + 셀러 변경 가능

### 원칙 3: 명확한 피드백

> 사용자가 한 행동의 결과를 즉시 확인할 수 있어야 한다.

- 모든 비동기 작업에 진행 상태 표시 (프로그레스 바, 스피너)
- 완료/실패 결과를 토스트 또는 상태 변경으로 즉시 피드백
- 알림 패널로 백그라운드 작업 결과 전달

**프로토타입 적용 사례:**
- URL 수집: 실시간 검증 패널 + 건별 진행 상태 (SC-02)
- AI 번역: 셰이머 애니메이션 → 결과 즉시 반영 (ED-02)
- 등록 프로그레스: 실시간 성공/실패 카운터 (RG-01)
- 토스트: 저장/삭제/번역 시작 등 모든 주요 액션에 즉각 피드백

### 원칙 4: 에러 시 해결책 제시

> 에러 메시지만 보여주지 말고, 다음 행동을 안내한다.

- 에러 발생 시 원인 + 해결 방법을 함께 표시
- "재시도" 같은 즉시 행동 버튼 제공
- 파괴적 에러는 확인 모달로 방지

**프로토타입 적용 사례:**
- URL 수집 실패: 에러 메시지 + "재시도" 링크 (SC-02)
- 등록 실패: 에러 유형별 해결 방법 안내 (RG-02)
- 역마진: "판매가 조정이 필요해요" + 편집 링크 (G-02 알림)
- 품절: "판매 일시중지가 필요해요" + 중지 버튼 (RG-02)

## UT 결과 반영 사항

### 1차 UT → 2차 수정
- 정보 과부하 해소: 단계별 정보 노출 강화
- 온보딩 인트로 애니메이션 추가 (순차 텍스트 표시)

### 2차 UT → 3차 수정
- 알림 패널 전면 재설계 (수집/번역/등록 알림 통합)
- 내 상품 화면에 모니터링 통합
- 설정을 모달에서 전체 페이지로 전환

### 3차 UT → 최종 반영
- **설정 패널 전면 개편**: 모달 → 전체 페이지 + 저장 피드백 + 환율 설정
- **모니터링 UX 개편**: 토글 UI, 상태별 탭 분류, "모니터링" vs "자동 조정" 구분
- **자동 수집 중복 처리**: 기수집 상품은 업데이트, 판매중 상품은 재수집 제외
- **크롬 익스텐션 온보딩 추가** (S-05)
- **"소싱" 용어 금지**: UI에서 "소싱" 대신 구체적 출처명(올리브영 등) 또는 "수집" 사용

## 측정 가능한 설계 목표

| 목표 | 기준 | 측정 방법 |
|------|------|-----------|
| 첫 등록 완주율 | 온보딩 시작 → 첫 등록 완료 80% 이상 | UT에서 완주 여부 관찰 |
| 화면당 핵심 액션 | 3개 이하 | 화면별 명세 검증 |
| AI 자동 완성 적용 | 번역/카테고리/가격 자동 처리 | 수동 입력 필드 수 최소화 |
| 에러 해결 안내율 | 모든 에러에 해결 방법 제시 | 에러 시나리오별 검증 |
| 피드백 즉시성 | 모든 주요 액션에 2초 이내 피드백 | 토스트/프로그레스 존재 확인 |

## UX 라이팅 규칙

### 용어 규칙
- **"소싱" 사용 금지** → "수집", "수집하기" 또는 구체적 출처명 사용
- "모니터링" → "가격·품절 확인" (사용자 친화적)
- "등록" → Qoo10에 상품 올리는 행위
- "편집" → 수집된 상품을 번역/가공하는 행위

### 톤 & 매너
- **친근하고 직접적**: "~해주세요", "~할 수 있어요", "~됐어요"
- **긍정 프레이밍**: "실패" 대신 "다시 시도해보세요"
- **구체적 안내**: "에러 발생" 대신 "쇼핑몰 접속이 일시적으로 불안정해요. 잠시 후 재시도해보세요"
- **줄임말/전문 용어 최소화**: "API Key" 같은 필수 용어는 툴팁으로 설명
